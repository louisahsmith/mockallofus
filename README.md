# mockallofus

A local mock database for developing *All of Us* analyses without the paid
Researcher Workbench.

The [`allofus`](https://github.com/roux-ohdsi/allofus) R package wraps access to
the *All of Us* Research Program data, which lives in Google BigQuery and is only
reachable from the (metered) Researcher Workbench. That makes iterative code
development — especially for students who rerun constantly — slow and expensive.

`mockallofus` builds a local [DuckDB](https://duckdb.org) database whose tables
and columns mirror the *All of Us* Curated Data Repository (OMOP CDM core tables
plus the *All of Us* extension and cohort-builder tables), populated with
**fully synthetic, randomly generated data**. You can then develop and test the
same `allofus` analysis code locally that you would run on the Workbench.

> The synthetic data is random and **does not reflect real *All of Us* data**
> (which would violate the data use agreement). Results are not meant to be
> realistic — the point is to make code run.

## Installation

`mockallofus` currently requires the development version of `allofus` from the
[`mock-duckdb-support`](https://github.com/roux-ohdsi/allofus/tree/mock-duckdb-support)
branch, which adds the backend-aware query execution that lets `allofus`
functions run against DuckDB rather than only BigQuery. (Once that work is
merged and released, the released `allofus` will work.)

```r
# install.packages("pak")
pak::pak("roux-ohdsi/allofus@mock-duckdb-support")  # required allofus branch
pak::pak("louisahsmith/mockallofus")                # installs duckdb too
```

`mockallofus` declares this requirement in its `Remotes:` field, so installing
it (or its dependencies, e.g. in CI) pulls the correct `allofus` branch
automatically.

## Usage

The only thing that changes versus a Workbench script is the connection call:
use `mock_aou_connect()` instead of `allofus::aou_connect()`. Everything
downstream is identical.

```r
library(allofus)
library(mockallofus)
library(dplyr)

con <- mock_aou_connect()   # builds the database on first run, then caches it

# ...the same code you'd run on the Workbench:
tbl(con, "person") |> count() |> collect()

cohort <- tbl(con, "person") |> select(person_id)

aou_concept_set(cohort, concepts = c(201826, 4193704), domains = "condition",
                output = "indicator", concept_set_name = "t2dm")

aou_survey(questions = c(1585838, 1586135),
           question_output = c("gender", "birthplace"))

aou_observation_period()
```

### Making your study's queries return data

A small default database means queries for *specific* concepts or eligibility
criteria can come back empty, which is harder to develop against than no data at
all. Rather than making the database enormous, **seed** the concepts your study
uses, at a prevalence you choose, so eligibility joins return partial cohorts:

```r
# guarantee these condition concepts exist for ~30% of participants
mock_seed_concept_set(con, concepts = c(201826, 4193704),
                      domain = "condition", prevalence = 0.3)

# guarantee responses to specific survey questions (real codebook concept ids)
mock_seed_survey(con, concept_ids = c(1585838, 1586135), prevalence = 0.6)
```

Seeding is deterministic (for a fixed `seed`) and attaches to existing
participants, so all downstream joins stay consistent.

## What works locally, and what doesn't

Because most `allofus` functions are pure `dbplyr` (which translates to DuckDB
automatically) and the package's internal query helper is backend-aware, the
large majority of workflows run unchanged.

**Works locally**

- `dplyr`/`dbplyr` on `tbl(con, ...)`, `aou_join()`, `aou_collect()`
- `aou_concept_set()`
- `aou_survey()` (with real codebook question concept ids)
- `aou_observation_period()`
- `aou_compute()`, `aou_create_temp_table()`
- `aou_sql()` — **provided the SQL is portable**. BigQuery-only constructs
  (`COUNTIF`, `APPROX_QUANTILES`, etc.) will not run; a few common ones
  (`CONTAINS_SUBSTR`, `REGEXP_CONTAINS`) are emulated via DuckDB macros.

**Workbench-only (clearly-labeled exceptions)**

- `aou_atlas_cohort()` — relies on BigQuery-dialect SQL generated by `SqlRender`.
- File/bucket management (`aou_ls_workspace()`, `aou_ls_bucket()`,
  `aou_workspace_to_bucket()`, `aou_bucket_to_workspace()`) — these use Google
  Cloud Storage, which has no local equivalent.
- `aou_session_info()` CDR-metadata details.
- User-written SQL that uses BigQuery-specific functions.

## How it works

- `build_mock_db()` creates every table in the *All of Us* data dictionary
  (from `allofus::aou_table_info`) and populates a core subset with synthetic
  rows. EHR-sourced rows are tagged in the `_ext` tables so EHR-based functions
  (like `aou_observation_period()`) behave correctly.
- `mock_aou_connect()` opens the DuckDB database, registers a few
  BigQuery-compatibility macros, attaches BigQuery-compatible `dbplyr`
  translations, and stores the connection in `getOption("aou.default.con")` and a
  schema name in `getOption("aou.default.cdr")` — exactly the options `allofus`
  reads.
- BigQuery-isms common in All of Us code are handled so pipelines run unchanged:
  the three-argument `date_diff(a, b, sql("day"))`, `as.Date()` of
  float-valued concatenated date parts (e.g. `"1994-5.0-28.0"`), and the
  `collect(page_size = ...)` download argument. More complex real-world
  pipelines (e.g. the HIPPS pregnancy algorithm) run end-to-end on the mock
  database after seeding their concept lists.
